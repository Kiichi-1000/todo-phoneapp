# ルーティン機能 仕様書（SSOT）

本書は「ルーティン」タブおよび統計連携の**単一の参照元（Single Source of Truth）**である。実装・DB・QAは本書の定義に従う。

---

## 1. 目的とスコープ

### 1.1 目的

- スケジュール・ワークスペースの ToDo と**独立**した「毎日の習慣」チェックリストを提供する。
- ユーザーが編集する**テンプレート**に基づき、日付ごとに「今日やったか」を記録する。
- 統計タブでルーティンの**完遂率**（定義は後述で1つに固定）を表示する。

### 1.2 非スコープ（明示）

- スケジュール（`schedules`）や ToDo（`todos`）との自動連動は行わない。
- リマインダー通知は本仕様では定義しない（将来拡張可）。

---

## 2. ユーザーストーリー

- **US-RTN-01**: ユーザーは「朝・日中・夜」の3ブロックに分かれたルーティン一覧を、選択した日付で確認できる。
- **US-RTN-02**: ユーザーは各項目をチェック／チェック解除し、その日の達成状況を記録できる。
- **US-RTN-03**: ユーザーはテンプレートを編集し、時間帯ごとに項目の追加・削除・並び替え・（任意）表示用の短いラベル調整ができる。
- **US-RTN-04**: ユーザーは統計画面で、期間内のルーティン完遂率および時間帯別の傾向を確認できる。

---

## 3. 用語・時間帯の定義

| 用語 | 説明 |
|------|------|
| **ルーティンテンプレート** | ユーザーごとの「習慣リストの型」。項目は時間帯（スロット）と並び順を持つ。 |
| **テンプレート項目** | テンプレート内の1行。チェック対象となる習慣の1つ。 |
| **日次チェック（完了記録）** | ある**暦日**において、あるテンプレート項目を「やった」と記録した事実。 |
| **アクティブ項目** | 集計・当日リストに含める項目。`is_active = true` のテンプレート項目（後述スキーマ）。 |

### 3.1 スロット（時間帯）

アプリ上の表示ラベルと内部値の対応は次のとおり固定する。

| 内部値 (`slot`) | 表示ラベル |
|-----------------|------------|
| `morning` | 朝 |
| `daytime` | 日中 |
| `evening` | 夜 |

**ビジネス上の意味**: 厳密な時刻帯の自動判定は行わない。ユーザーが「どのブロックに置くか」で分類する。将来、時刻ヒントを付与する拡張は本仕様の範囲外。

## 4. 画面ワイヤー（箇条書き）

### 4.1 ルーティンタブ（`app/(tabs)/routine.tsx` 想定）

- **ヘッダー**
  - 表示中の暦日（デフォルトは「今日」）。
  - 前日／翌日への切替（ワークスペースと同様の「1日単位」体験）。
  - （任意）カレンダーまたは日付ピッカーで任意日へジャンプ。
- **本文**
  - セクション3つ: 朝 / 日中 / 夜（各 `morning` / `daytime` / `evening`）。
  - 各セクション内: 当該スロットの**アクティブ**テンプレート項目を `sort_order` 順にリスト表示。
  - 各行: チェックボックス、表示テキスト（`title` または短縮ラベル）、（任意）並びの視覚的ヒント。
  - セクションは折りたたみ可能にしてもよい（実装詳細は任意）。
- **テンプレート編集への導線**
  - ツールバーまたはFAB等で「テンプレートを編集」を開く。
- **空状態**
  - テンプレートに項目がない場合は、編集への誘導メッセージを表示する。

### 4.2 テンプレート編集（モーダルまたは別画面）

- スロットごとにリストを表示。
- 各スロットで: 追加、削除、並び替え（ドラッグまたは↑↓）、項目タイトル（および任意の短い表示名）の編集。
- 保存時: **テンプレートおよび項目の永続化**（翌日以降も同じテンプレート定義を参照）。
- **非アクティブ化**（推奨）: 項目を「一覧から隠すが履歴は残したい」場合は `is_active = false` とする（削除とは別操作としてもよい）。

---

## 5. 日次データモデル（採用方針: 案B）

当日の画面表示は常に**現在のテンプレート項目**を基準とし、チェックの有無は別テーブルで保持する（テンプレと表示を LEFT JOIN 的に合成）。

- **メリット**: マイグレーションとクエリが単純。統計は「完了数 ÷ 期待チェック数」で集計しやすい。
- **注意**: テンプレ項目削除後、過去の `routine_completions` は `item_id` が孤児になり得る（エッジケース参照）。

---

## 6. データベース案（Supabase / PostgreSQL）

既存の `schedules` 等と同様に **`user_id` を付与**し、RLS はプロジェクト方針に合わせて設定する（本書は論理スキーマを定義する）。

### 6.1 列挙型

```sql
-- 例（マイグレーションで定義）
CREATE TYPE routine_slot AS ENUM ('morning', 'daytime', 'evening');
```

### 6.2 テーブル: `routine_templates`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | `uuid` PK | |
| `user_id` | `uuid` NOT NULL | 認証ユーザーと整合 |
| `updated_at` | `timestamptz` | 更新日時 |

**備考**: 当面はユーザーあたり1行でよい。将来「複数テンプレ」が必要なら `name` 等を追加する拡張とする。

### 6.3 テーブル: `routine_template_items`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | `uuid` PK | |
| `template_id` | `uuid` NOT NULL FK → `routine_templates.id` | |
| `slot` | `routine_slot` NOT NULL | 朝 / 日中 / 夜 |
| `sort_order` | `int` NOT NULL | 同一 `template_id` + `slot` 内の表示順（小さいほど上） |
| `title` | `text` NOT NULL | 主表示テキスト |
| `short_label` | `text` NULL | 任意。省略時は `title` を表示 |
| `is_active` | `boolean` NOT NULL DEFAULT true | false は当日リスト・集計の「期待項目数」から除外 |
| `created_at` | `timestamptz` | |
| `updated_at` | `timestamptz` | |

**インデックス（推奨）**: `(template_id, slot, sort_order)`

### 6.4 テーブル: `routine_completions`

| カラム | 型 | 説明 |
|--------|-----|------|
| `id` | `uuid` PK | （任意。`user_id, item_id, date` でユニークなら省略も可） |
| `user_id` | `uuid` NOT NULL | |
| `item_id` | `uuid` NOT NULL FK → `routine_template_items.id` | |
| `date` | `date` NOT NULL | 暦日（ユーザーのローカル日付をアプリで YYYY-MM-DD として保存する想定） |
| `completed_at` | `timestamptz` NOT NULL | チェックした時刻 |

**制約**: `UNIQUE (user_id, item_id, date)` — 同一日・同一項目は1完了行のみ。

**インデックス（推奨）**: `(user_id, date)`

### 6.5 アプリ型定義

実装時は `types/database.ts` の `Database` に上記テーブルの `Row` / `Insert` / `Update` / `Relationships` を追加し、Supabase クライアントから型安全に参照する。

## 7. フロー仕様

### 7.1 日次チェックフロー

1. ユーザーが日付 `D` を選択する。
2. アプリは `routine_templates` を `user_id` で取得（なければ空テンプレまたは初回作成）。
3. `routine_template_items` を `template_id` で取得し、`is_active = true` のみ、`slot` ごとに `sort_order` で並べる。
4. 各 `item_id` について `routine_completions` に `(user_id, item_id, date = D)` があるかでチェック状態を決定。
5. ユーザーがチェック ON: `routine_completions` に INSERT（`completed_at = now()`）。
6. ユーザーがチェック OFF: 該当行を DELETE（または論理削除は採用しない — 物理DELETEでよい）。

### 7.2 テンプレート編集フロー

1. 編集画面で項目の追加・更新（タイトル・短縮ラベル・スロット・並び・`is_active`）を行う。
2. 保存時に `routine_template_items` を UPSERT / DELETE で整合させる。`routine_templates.updated_at` を更新する。
3. **過去日の画面**: 原則として「当時のスナップショット」は保持しない。過去日を開いた場合も**現在のテンプレート**に対するチェック有無を表示する（案Bの意味するところ）。

## 8. クエリ例（PostgREST / supabase-js 想定）

### 8.1 某日 `D` のリスト＋チェック状態

1. `routine_templates` を `user_id` で単一行取得。
2. `from('routine_template_items').select('*').eq('template_id', id).eq('is_active', true).order('slot').order('sort_order')`
3. `item_id` の配列を作り、`from('routine_completions').select('item_id').eq('user_id', uid).eq('date', D).in('item_id', ids)` で完了集合を取得。クライアントでマージ。

（単一SQLで書く場合は RPC や view でもよいが、本仕様では必須としない。）

### 8.2 チェックのトグル

- ON: `insert({ user_id, item_id, date: D, completed_at: new Date().toISOString() })` — 競合時は UNIQUE 違反を無視または事前 SELECT。
- OFF: `delete().match({ user_id, item_id, date: D })`

## 9. 統計（固定定義）

統計画面（`app/(tabs)/statistics.tsx` またはサブコンポーネント）では、次の**1つの定義**を用いる。

### 9.1 期間

プリセット: **今日** / **今週** / **今月**（いずれも暦日ベース。週の開始日はアプリ既存の統計と揃える — 例: 月曜始まり）。

対象日集合を \( \mathcal{D} \)、各日 \( d \in \mathcal{D} \) とする。

### 9.2 1日あたりの「期待チェック数」

日 \( d \) において、その時点のテンプレートで**アクティブ**な項目数を \( n_d \) とする。

**実装指針**: 案Bでは「その日のテンプレスナップショット」がないため、**集計期間中は「集計実行時点のアクティブ項目集合」を全日に適用する**簡易方式を採用する。

- \( I \) = 集計時点で `is_active = true` である全 `routine_template_items` の集合。
- 全日で \( n = |I| \)（テンプレに項目が無い場合は \( n = 0 \)）。

### 9.3 1日あたりの「実際の完了数」

日 \( d \) において、`routine_completions` の行数で、かつ `item_id \in I` であるものだけを数える（削除済み項目への孤児完了は除外）。これを \( c_d \) とする。

### 9.4 期間全体のルーティン完遂率（メイン指標）

\[
\text{RoutineCompletionRate} =
\begin{cases}
0 & \text{if } |\mathcal{D}| = 0 \text{ or } n = 0 \\[6pt]
\dfrac{\sum_{d \in \mathcal{D}} c_d}{|\mathcal{D}| \cdot n} \times 100\% & \text{otherwise}
\end{cases}
\]

解釈: **「期間内の各日について、（その日に記録された有効項目への完了数）を足し、（日数 × 現アクティブ項目数）で割った値」**。100% は、毎日すべてのアクティブ項目が完了している状態。

### 9.5 時間帯別完遂率（サブ指標）

スロット \( s \in \{\text{morning}, \text{daytime}, \text{evening}\} \) ごとに、アクティブ項目集合を \( I_s \)（`slot = s`）、\( n_s = |I_s| \) とする。

\[
\text{Rate}_s =
\begin{cases}
\text{N/A または非表示} & n_s = 0 \\
\dfrac{\sum_{d \in \mathcal{D}} c_{d,s}}{|\mathcal{D}| \cdot n_s} \times 100\% & \text{otherwise}
\end{cases}
\]

ここで \( c_{d,s} \) は、日 \( d \) にスロット \( s \) の項目に対する完了数（`item_id \in I_s` かつ当日 `completions` あり）。

**UI**: メインの1数値に加え、朝・日中・夜の3つの数値またはバーで表示する。

## 10. エッジケースと方針

| 状況 | 方針 |
|------|------|
| テンプレ未作成 | 初回アクセスで `routine_templates` を1行作成するか、項目0件の空表示とする（実装でどちらか選択し、一貫させる）。 |
| アクティブ項目が0件 | 完遂率は定義上 0% または「データなし」表示。ゼロ除算を避ける。 |
| テンプレ項目を**削除**（行DELETE） | `routine_completions` に残った `item_id` は**集計から除外**（JOIN 不能な孤児レコードになり得る）。履歴を厳密に残す必要が出たら、別タスクでクリーンアップジョブまたはソフトデリートを検討。 |
| 項目を**非アクティブ**化 | 以降の当日リストと統計の \( I \) から除外。過去に付いていた完了行はDBに残るが、上記集計式では `item_id \in I` でフィルタするため**過去日を遡っても「現アクティブ集合」基準**となる点に注意（簡易方式のトレードオフ）。 |
| タイムゾーン | `date` はアプリが決めた暦日（通常は端末ローカル）で保存。ユーザーが旅行でTZが変わった場合の揺れは許容し、詳細は既存アプリの日付扱いに合わせる。 |
| 重複チェック | DB の UNIQUE(`user_id`,`item_id`,`date`) で防ぐ。 |
